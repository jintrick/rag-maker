# Issue: `ragmaker-github-fetch` における GitHub リポジトリ取得失敗時のリトライロジックの修正

## 背景と問題点
`src/ragmaker/tools/github_fetch.py` の `github_fetch` 関数において、Shallow Clone (`depth=1`) が失敗した際のリトライ処理（Full Clone）に欠陥がある。
1回目のクローン失敗時に作成された `.git` ディレクトリ等の残留物が原因で、2回目のクローンが `fatal: destination path '...' already exists and is not an empty directory.` というエラーで失敗し、本来の失敗原因が隠蔽されてしまう。

## 解決策
- リトライ前に `clone_dir` を削除・再作成するロジックを追加する。
- 1回目の失敗時のエラーメッセージをログまたは例外に含め、デバッグを容易にする。
- 修正対象: `src/ragmaker/tools/github_fetch.py`

## 実装内容詳細
### 1. `github_fetch` 関数の修正
- `Repo.clone_from(repo_url, clone_dir, **kwargs)` の最初の試行が失敗した際の `except` ブロック内で以下を実行する：
  - `shutil.rmtree(clone_dir, ignore_errors=True)` により、失敗したクローンの中間成果物を完全に削除する。
  - `clone_dir.mkdir()` を実行し、空のディレクトリを再準備する。
  - 2回目の `Repo.clone_from` 実行時、1回目の例外 `e` の内容をログ (`logger.warning`) に含める。

### 2. 例外メッセージの改善
- リトライも失敗した場合の `RuntimeError` に、1回目の例外 (`e`) と2回目の例外 (`final_exception`) の両方の情報が含まれるようにする。
