## 経緯
v1.0.1 の実装過程において、`issue-crafter` サブエージェントが生成した設計案に、すでに v1.0.0 で廃止済みであった `catalog.json` や `enrich_discovery` への依存が再混入（デグレード）し、実行時に致命的な矛盾が発生した。本 Issue では、このデグレードを完全に解消し、当初の目的である「カタログレス・記述的命名規則」を維持したまま、ワークフローの強制力を最大化した。

## 現状の問題点
1. **カタログ依存の再混入（解消済み）**: `rag.md` に存在していた `catalog.json` や関連ツールの使用手順を完全に削除した。
2. **ワークフローの脆弱性**: 工程の省略、内容未確認の命名、環境制限による停止を物理的に防ぐための強力なガードレール（`[MUST: NO SKIPPING]`, Resilient Triage等）を統合した。

## 解決策の概要
`rag.md` を「完全カタログレス版」としてゼロから再構築し、そこに v1.0.1 で定義した強力なワークフローガードレールを統合した。

## 解決策の詳細
1. **ステップ 1**: 取得ツール（`http_fetch` 等）の出力を `catalog.json` に保存する記述を削除。ツールが自動でフロントマターを付与することを明記。
2. **ステップ 1.5**: カタログの精査ではなく、ディレクトリ（`.tmp/cache/`）の精査に変更。`rm` 失敗時の退避先を `.tmp/trash/` とする。
3. **ステップ 2**: カタログの更新（`enrich_discovery`）を削除し、`read_file` -> `move_file` の Atomic サイクルを定義。具体的命名例（Good/Bad）をインライン化。
4. **ステップ 3**: カタログへの全体メタデータ登録（`entry_discovery`）を削除。
5. **最終メッセージ**: `/ask` コマンドへの言及を完全に削除。
