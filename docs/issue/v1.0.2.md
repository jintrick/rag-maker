# Issue v1.0.2 - RAGMaker: Correcting Workflow Regression & Finalizing Enforcement

## 前提知識
- .gemini/commands/rag.md
- docs/guidelines/descriptive_naming_guidelines.md

## 経緯
v1.0.1 の実装過程において、`issue-crafter` サブエージェントが生成した設計案に、すでに廃止済みであった `catalog.json` や `enrich_discovery` への依存が再混入（デグレード）していた。本 Issue では、このデグレードを修正し、当初の目的である「カタログレス・記述的命名規則」を維持したまま、ワークフローの強制力を高める。

## 現状の問題点
1. **カタログ依存の再混入**: `rag.md` に `catalog.json` や関連ツールの使用手順が誤って記載されている。
2. **ワークフローの脆弱性**: 以前の失態と同様、工程の省略や曖昧な命名を防ぐための制約が不十分なままになっている。

- [x] 「Atomic な処理（読込後、即命名・移動）」が明記されている。

## 解決策の概要
`rag.md` を「完全カタログレス版」として再構築し、そこに v1.0.1 で定義した強力なワークフローガードレールを統合する。

## 解決策の詳細
実装担当者は、`rag.md` の内容が以下の要件を満たしていることを確認（または修正）すること。
1. **ステップ 1**: 取得ツール（`http_fetch` 等）の出力を `catalog.json` に保存する記述を削除。ツールが自動でフロントマターを付与することを明記。
2. **ステップ 1.5**: カタログの精査ではなく、ディレクトリ（`.tmp/cache/`）の精査に変更。`rm` 失敗時の退避先を `.tmp/trash/` とする。
3. **ステップ 2**: カタログの更新（`enrich_discovery`）を削除し、`read_file` -> `move_file` の Atomic サイクルを定義。具体的命名例（Good/Bad）をインライン化。
4. **ステップ 3**: カタログへの全体メタデータ登録（`entry_discovery`）を削除。
5. **最終メッセージ**: `/ask` コマンドへの言及を削除。
