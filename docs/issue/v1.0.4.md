# Issue v1.0.4 - RAGMaker: Multi-Agent Organization Audit & Classification Refinement

## 1. 目的 (Objective)
RAGMaker v1.0.x におけるドキュメント分類の精度をさらに向上させ、特に `appendix` への不適切な分類（Example や Workflow の混入）を完全に排除する。
「ガイドラインの厳格化」と「第三者エージェント（`generalist`）による組織化監査ステップ」を導入することで、ナレッジベースの品質を自律的に洗練させる。

## 2. 解決すべき問題点 (Root Cause Analysis)
- **分類の曖昧さ**: 現状のガイドラインでは、LLM が `Example` や `Setup` を「補足情報 (Appendix)」と解釈しやすい。
- **単一エージェントの限界**: 整理を行ったエージェント自身が「最終チェック」も行っているため、自身の誤判定（Bias）に気づきにくい。
- **命名の具体性不足**: 具体的なユースケース（Workflow）が含まれるファイルが、抽象的な名前で `appendix` に埋もれてしまう。

## 3. 完了条件 (Acceptance Criteria)
- [x] `docs/guidelines/descriptive_naming_guidelines.md` において、`guide` と `appendix` の境界が明確（Example/Workflow は `guide` 強制）に再定義されている。
- [x] `.gemini/commands/rag.md` に、整理の最終段階で `generalist` サブエージェントを呼び出し、構成を監査するステップ（2.6）が追加されている。
- [x] `generalist` による指摘があった場合、主担当エージェントが適切に修正（再分類・改名）を行うループが指示されている。
- [x] `appendix` には「変更履歴、利用規約、割当制限、用語集」等のメタデータのみが残り、学習価値の高いドキュメントは `guide` または `reference` に集約されるようになる。

## 4. 解決策の詳細 (Solution Detail)

### 4.1. ガイドラインの修正
- **`guide`**: 「How to achieve X」という全ての能動的ドキュメント（Examples, Workflows, Tutorials, Recipes）を包含。
- **`appendix`**: 「Static metadata」に限定。履歴、ライセンス、用語集、FAQ（手順を含まない一問一答）。
- **禁止事項**: `Example` や `Workflow` キーワードを含むファイルを `appendix` に入れることを厳禁とする。

### 4.2. `rag.md` への「監査ステップ (2.6)」の導入
- **プロセス**: 
    1.  `generalist` サブエージェントを呼び出す。
    2.  `.tmp/cache/` 内の全ファイルリストを渡し、「[ガイドライン] に基づき、特に `appendix` に不適切なファイルが混じっていないか」をチェックさせる。
    3.  `generalist` の指摘に基づき、必要なら `move_file` で修正を実行。

## 5. 修正対象資産 (Affected Files)
- `docs/guidelines/descriptive_naming_guidelines.md`
- `.gemini/commands/rag.md`

## 6. テスト手順 (Verification Steps)
1. `gemini /rag` で実際に KB を構築し、ステップ 2.6 で `generalist` が起動することを確認。
2. 構築後の `appendix/` ディレクトリに、実装例（Example）や手順書（Workflow）が含まれていないことを確認。
3. `guide/` にそれらが正しく配置されていることを確認。
