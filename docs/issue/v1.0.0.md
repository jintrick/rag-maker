# Issue v1.0.0 - RAGMaker: Catalogless & Descriptive Naming (Surgical Update)

## 1. 目的 (Objective)
現在の `rag.md` が持つ「一時ディレクトリ（`.tmp/cache/`）での一括処理と、最終配置先への一括移動」という堅牢なワークフローを維持したまま、メタデータ管理を `catalog.json`（中央集権）から、**「記述的命名規則」と「ファイル内フロントマター」**（自律分散）へと切り替える。

## 2. 完了条件 (Acceptance Criteria)
- [ ] **既存フローの堅持**: `rag.md` の全体構造（ステップ 0:準備 〜 ステップ 4:最終配置）を維持し、`.tmp/cache/` を作業場とする設計を変更していない。
- [ ] **Phase 2 (Enrich) の置換**: `rag.md` の「ステップ 2. ドキュメントのエンリッチ」を、`enrich_discovery` の呼び出しではなく、**[命名規則ガイドライン](docs/guidelines/descriptive_naming_guidelines.md) に基づくファイルのリネーム（`move_file`）** 処理へと、既存のループ構造を維持したまま変更している。
- [ ] **フロントマターの自動挿入**: `http_fetch` および `github_fetch` ツールが、Markdown 保存時に YAML フロントマター（`source_url`, `original_title`, `fetched_at`）を先頭に自動付加している。
- [ ] **カタログ依存ツールの廃止**: `catalog.json` を操作するための `enrich_discovery`, `entry_discovery` を物理削除し、`discovery.json` および `pyproject.toml` から定義を抹消している。
- [ ] **`/ask` コマンドの廃止**: 専用コマンドを介さず、エージェントが標準ツール（`list_directory`, `read_file`）を用いて自律的に情報を探索・回答できる環境を確立している。

## 3. 解決策の詳細 (Solution Detail)

### 3.1. 収集ツールの改修 (Python)
AIエージェントのターンを消費せず、**Atomic** にメタデータを付与するため、以下のツールを改修する。
- **対象**: `src/ragmaker/tools/http_fetch.py`, `github_fetch.py`
- **内容**: 保存する Markdown 文字列の先頭に、以下の YAML フロントマターを固定値として挿入するロジックを追加。
```yaml
---
source_url: {{url}}
original_title: {{title}}
fetched_at: {{timestamp}}
---
```

### 3.2. ワークフロー（`rag.md`）の外科的修正
全体のフロー構造は変えず、ステップ 2 と 3 の「内容」のみを差し替える。

- **ステップ 2 (ドキュメントの整理)**:
    - 現在の「10件ずつのループ」と「直接読み込み」は維持。
    - **変更前**: `title` と `summary` を生成し、`enrich_discovery` で `catalog.json` を更新。
    - **変更後**: [記述的命名規則ガイドライン] に基づき、移動先パス（例: `guide/ragmaker_setup.md`）を決定。`.tmp/cache/` 内で **`move_file` を実行してリネーム** する。
- **ステップ 3 (メタデータ登録)**: 
    - **削除。** カタログレス化により、ナレッジベース全体への `title`/`summary` 登録は不要となる。
- **ステップ 4 (最終配置)**: 
    - **現状維持。** 整理が終わった `.tmp/cache/` を、そのまま `{{kb_root}}/cache/` へ `move_file` で一括移動する。

4.  ファイルを開き、先頭に正しいフロントマターが記載されていることを確認。
5.  [この項目はユーザーが自らテストする] エージェントに対し、特定のトピックについて質問し、エージェントが `list_directory` でファイルを特定し、`read_file` で読み込んで回答することを確認。
