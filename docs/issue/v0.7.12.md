## 経緯
`ragmaker-install-kb` ツールにおいて、特定の条件下でディレクトリ構造が不必要に入れ子になったり、ソースディレクトリの指定方法によってファイルが正しくコピーされない問題が報告された。特に、ソースとして `cache` ディレクトリ自体を指定した場合や、ターゲットディレクトリの直下に内容を展開したい場合に、ユーザーの意図した構造にならない。

## 現状の問題点
1. **強制的なサブディレクトリ化**: `merge` が `false`（デフォルト）の場合、必ず `--source` で指定したディレクトリ名がターゲット内に作成される。ターゲット直下に展開する手段がない。
2. **cache ディレクトリの二重化 (`cache/cache/`)**: ソースとして `.../cache` ディレクトリを指定した場合、内部でさらに `cache` サブディレクトリを探そうとする、あるいはターゲット側で `cache/cache` という構造が作られてしまう。
3. **--merge フラグの不備**: ソースが標準的な KB 構造（ルートに `cache/` がある状態）でない場合、`--merge` を指定してもファイルがコピーされないことがある。

## 解決策の概要
`install_knowledge_base` 関数に `flatten` オプションを追加し、`_install_merged` 内部でのソース構造の判定ロジックを強化する。また、カタログ内のパス計算ロジックをより堅牢なものに修正する。

## 解決策の詳細
- `src/ragmaker/tools/install_kb.py` の修正：
  - `main` 関数で `--flatten` 引数（`action="store_true"`）を追加。
  - `install_knowledge_base` 関数を、`merge` または `flatten` が `True` の場合に `_install_merged` を直接呼ぶように変更。
  - `_install_merged` 内のループで各 `source_root` に対して以下の判定を行う：
    - `(source_root / "cache").is_dir()` が真なら `source_cache = source_root / "cache"`、`source_base = source_root`。
    - そうでなく、`source_root.name == "cache"` なら `source_cache = source_root`、`source_base = source_root.parent`。
    - それ以外（フラットなディレクトリ）なら `source_cache = source_root`、`source_base = source_root`。
  - パス正規化の修正：
    - `doc_path` が既に `cache/` で始まっている場合や、`abs_source_path` を `source_cache` からの相対パスで再計算することで、確実に `cache/filename` の形式にする。
- `discovery.json` の更新：
  - `install_kb` ツールのパラメータに `flatten` を追加。
